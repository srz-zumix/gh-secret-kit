# Migration with --dst-token (Delegated Destination Access)

This guide explains how to use `--dst-token` when the operator running
gh-secret-kit does not personally hold a token for the destination host.

## When to Use --dst-token

Use `--dst-token` when:

- The operator does not have `gh auth login` credentials for the destination
  host (e.g., a teammate or admin provides a PAT stored as a repository secret)
- The self-hosted runner is not authenticated to the destination host
- You want the migration workflow to use a specific PAT rather than the
  runner's default `gh` authentication

If you are authenticated to both hosts via `gh auth login`, you do **not**
need `--dst-token`.

## Critical: --dst-token Means Different Things

`--dst-token` has **different semantics** depending on the subcommand:

| Subcommand | --dst-token value | How it is used |
| --- | --- | --- |
| `create`, `all` | **Secret name** (e.g., `DST_PAT`) | Referenced as `${{ secrets.DST_PAT }}` in the generated workflow |
| `check` | **Token value** (e.g., `ghp_xxx...`) | Used directly by the CLI to call the destination API |

### For create / all: Pass a Secret Name

```bash
# CORRECT: pass the NAME of the secret, not the token value
gh secret-kit migrate repo all \
  -s owner/source-repo \
  -d enterprise.internal/owner/dest-repo \
  --dst-token DST_PAT

# WRONG: do NOT pass the actual token value
# gh secret-kit migrate repo all ... --dst-token ghp_xxxxx  # Wrong!
```

The workflow YAML generated by `create` / `all` will contain:

```yaml
env:
  GH_TOKEN: ${{ secrets.DST_PAT }}
```

This means the secret named `DST_PAT` must exist on the **source repository**
and contain a valid PAT for the destination host.

### For check: Pass the Token Value

```bash
# CORRECT: pass the actual token value
gh secret-kit migrate repo check \
  -s owner/source-repo \
  -d enterprise.internal/owner/dest-repo \
  --dst-token ghp_xxxxx
```

The `check` command runs locally (not via a workflow), so it needs the actual
token to query the destination API.

## Setup: Register the Destination PAT as a Source Secret

Before running migration with `--dst-token`, you must store the destination
PAT as a repository secret on the **source** repository:

```bash
# Register the destination host PAT as a secret on the source repo
gh secret set DST_PAT -R owner/source-repo
# (paste the destination host PAT when prompted)
```

The secret name (e.g., `DST_PAT`) is what you pass to `--dst-token` in
`create` and `all` commands.

## Full Workflow Example

### Repository Secret Migration

```bash
# Step 0: Store destination PAT on the source repo
gh secret set DST_PAT -R owner/source-repo

# Terminal 1: Start runner on source
gh secret-kit migrate runner setup -R owner/source-repo

# Terminal 2: Migrate using --dst-token (secret name)
gh secret-kit migrate repo all \
  -s owner/source-repo \
  -d enterprise.internal/owner/dest-repo \
  --dst-token DST_PAT

# Verify (uses token value for local check)
gh secret-kit migrate repo check \
  -s owner/source-repo \
  -d enterprise.internal/owner/dest-repo \
  --dst-token "$(gh auth token --hostname enterprise.internal)"

# Clean up
gh secret-kit migrate runner teardown -R owner/source-repo
```

### Environment Secret Migration

```bash
# Step 0: Store destination PAT on the source repo
gh secret set DST_PAT -R owner/source-repo

# Terminal 1: Start runner
gh secret-kit migrate runner setup -R owner/source-repo

# Terminal 2: Migrate environment secrets
gh secret-kit migrate env all \
  -s owner/source-repo \
  -d enterprise.internal/owner/dest-repo \
  --src-env production \
  --dst-env production \
  --dst-token DST_PAT

# Verify
gh secret-kit migrate env check \
  -s owner/source-repo \
  -d enterprise.internal/owner/dest-repo \
  --src-env production \
  --dst-env production \
  --dst-token "$(gh auth token --hostname enterprise.internal)"

# Clean up
gh secret-kit migrate runner teardown -R owner/source-repo
```

### Organization Secret Migration

```bash
# Step 0: Store destination PAT on a repo in the source org
gh secret set DST_PAT -R source-org/some-repo

# Terminal 1: Start runner
gh secret-kit migrate runner setup source-org

# Terminal 2: Migrate org secrets
gh secret-kit migrate org all \
  -s source-org/some-repo \
  -d enterprise.internal/dest-org \
  --dst-token DST_PAT

# Verify
gh secret-kit migrate org check \
  -s source-org \
  -d enterprise.internal/dest-org \
  --dst-token "$(gh auth token --hostname enterprise.internal)"

# Clean up
gh secret-kit migrate runner teardown source-org
```

## Step-by-Step (Without all)

When using individual steps, only `create` requires `--dst-token`:

```bash
gh secret set DST_PAT -R owner/source-repo

gh secret-kit migrate repo init -s owner/source-repo

# Only create needs --dst-token (it generates the workflow referencing the secret)
gh secret-kit migrate repo create \
  -s owner/source-repo \
  -d enterprise.internal/owner/dest-repo \
  --dst-token DST_PAT

gh secret-kit migrate repo run -s owner/source-repo --wait

# check needs the actual token value
gh secret-kit migrate repo check \
  -s owner/source-repo \
  -d enterprise.internal/owner/dest-repo \
  --dst-token "$(gh auth token --hostname enterprise.internal)"

gh secret-kit migrate repo delete -s owner/source-repo
```

## Common Mistakes

| Mistake | What happens | Fix |
| --- | --- | --- |
| Passing token value to `create`/`all` | Token is embedded in workflow YAML as a string literal (insecure and won't work) | Pass the secret **name** (e.g., `DST_PAT`) |
| Passing secret name to `check` | CLI tries to use the literal string as a token and gets 401 | Pass the actual **token value** |
| Forgetting to register the secret | Workflow runs but `${{ secrets.DST_PAT }}` resolves to empty | Run `gh secret set DST_PAT -R owner/source-repo` first |
| Using wrong secret scope | Secret not accessible to the workflow | Ensure the secret is set as a **repository secret** on the source repo |

## Security Notes

- The token value is never embedded in the workflow YAML. Only the secret name
  appears in the generated workflow.
- The workflow reads the token at runtime via `${{ secrets.<name> }}`.
- Secret values are never written to disk on the runner.
- After migration, consider deleting the destination PAT secret from the source
  repository:
  ```bash
  gh secret delete DST_PAT -R owner/source-repo
  ```
- The destination PAT should have minimal required scopes for the migration
  target (e.g., `repo` scope for repository secrets, `admin:org` for org
  secrets).
